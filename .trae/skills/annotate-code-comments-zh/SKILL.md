# 中文代码注释模板

把代码文件按统一的中文注释风格补充说明，适用于用户要求：

- “详细注释一下这个文件/这个模块”
- “逐个注释每个函数”
- “说明每个函数的定位、作用、调用关系”
- “按某种固定风格统一整理注释”

## 要求

输出的注释应满足这几个要求：

- 文件头说明模块定位、职责、主要提供内容、典型调用链
- 函数头只写高层信息，不把“步骤流程”堆在函数头里
- 函数体内部用就近注释解释关键步骤
- 不改业务逻辑，只补注释或调整注释位置
- 优先中文注释，如果存在英文注释则翻译后融入新增注释，注意专业术语保留必要英文原词

## 注释规则

### 1. 文件头注释

在文件顶部使用块注释，说明：

- 文件定位
- 核心职责
- 提供的主要能力
- 典型调用链

示例：

```ts
/**
 * 资源加载器模块
 *
 * 文件定位：coding-agent 的统一资源加载与管理层。
 *
 * 功能概述：
 * - 集中管理扩展、技能、提示模板、主题和上下文文件
 * - 合并用户级、项目级和 CLI 指定资源
 *
 * 调用链路：
 *   应用启动 → DefaultResourceLoader.reload() → packageManager.resolve() → 各资源加载器
 */
```

### 2. 函数头注释

函数头只写高层语义，不写详细步骤列表。

优先包含：

- 作用
- 定位
- 被谁调用（本代码以及项目其余代码都要考虑）
- 调用了谁（如果为空可以不写）
- 参数和返回值

函数头注释复杂度要与函数内代码量匹配，简单的函数只需一句话总结即可。

示例：

```ts
/**
 * 从指定目录中加载第一个找到的上下文文件。
 *
 * 定位：模块内部辅助函数，为 loadProjectContextFiles() 提供单目录级别的上下文发现能力。
 *
 * 被谁调用：
 *   - loadProjectContextFiles()
 *
 * 调用了谁：
 *   - node:fs.existsSync()
 *   - node:fs.readFileSync()
 *   - node:path.join()
 *
 * @param dir 要搜索的目录绝对路径
 * @returns 找到时返回 { path, content }，否则返回 null
 */
```

### 3. 函数体内部步骤注释

“步骤流程”必须直接写在函数体内部，放在对应代码块前，不放在函数头里。

要求：

- 按执行阶段分段
- 注释尽量贴近代码
- 解释“为什么这样做”，不只是复述代码字面意思
- 简单 getter 只写最少注释

示例：

```ts
function resolvePromptInput(input: string | undefined, description: string): string | undefined {
	// 空输入直接视为未提供。
	if (!input) {
		return undefined;
	}

	// 若输入本身是一个存在的文件路径，则优先读取文件内容。
	if (existsSync(input)) {
		try {
			return readFileSync(input, "utf-8");
		} catch (error) {
			// 文件可见但读取失败时，退回原始输入，避免彻底丢失提示词。
			console.error(chalk.yellow(`Warning: Could not read ${description} file ${input}: ${error}`));
			return input;
		}
	}

	// 不存在的路径按纯文本提示词处理。
	return input;
}
```

## 执行原则

- 先通读文件，再决定注释密度
- 大函数按阶段分块注释
- 小函数只补必要上下文
- 不给每一行都加注释
- 不写“把值赋给变量”这类废话注释
- 不改变现有逻辑、控制流、类型和行为
- 若文件已有注释，优先统一风格，而不是机械叠加

## 推荐工作流

1. 先识别模块职责和调用链
2. 给文件头补模块级注释
3. 给导出函数、核心类、关键私有方法补函数头注释
4. 把原本写在函数头里的“步骤说明”下沉到函数体内部
5. 检查注释是否贴近实现、是否冗余
6. 运行诊断，确认没有因注释格式引入问题

## 可直接复用的提示模板

当用户要你按这套方式注释代码时，优先使用下面这段提示：

```text
请按以下规范为目标代码补充中文注释，不要修改业务逻辑：

1. 文件头补模块级注释，说明：
- 文件定位
- 核心职责
- 主要提供内容
- 典型调用链

2. 每个重要函数/方法的函数头补充：
- 作用
- 定位
- 被谁调用
- 调用了谁
- 参数 / 返回值（需要时）

3. 不要把“内部步骤”“执行流程”写在函数头注释里。
请把步骤说明直接写进函数体内部，并贴近对应代码块，按执行阶段分段说明。

4. 注释风格要求：
- 中文为主，必要术语保留英文
- 简单函数少写，复杂函数多写
- 不写废话注释
- 不改逻辑，只补注释

5. 如果文件已有注释，请统一成上述风格，而不是重复堆叠。
```

## 交付标准

完成后应满足：

- 注释能帮助读者理解模块职责和调用链
- 读函数时可以一边看代码一边看步骤注释
- 头部注释不冗长，函数体注释不碎片化
- 文件通过基础诊断检查
